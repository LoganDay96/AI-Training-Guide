Best Practices

Guidelines for building better projects with AI assistance. Beginner-friendly with real-world analogies.

Streamlit vs JavaScript: When to Use What Web Development
Streamlit is perfect for quick dashboards and data analysis. JavaScript/React is better for interactive websites that need to feel fast and responsive.

Use Streamlit when:

  • Building data dashboards or reports
  • Creating internal tools just for your team
  • Prototyping ideas quickly (hours, not days)
  • Working primarily with Python and data (Pandas, charts)
  • You don't need real-time interactivity

Use JavaScript/React when:

  • Building customer-facing websites
  • Need instant feedback (no page reloads)
  • Complex interactivity (drag-and-drop, animations)
  • Mobile-first responsive design is critical
  • Integration with existing web applications
Analogy: Streamlit is like a food truck - fast to set up, great for specific use cases. JavaScript is like a full restaurant - more work to build, but handles anything.
Start Simple, Add Complexity Later Web Development
Always start with the simplest solution that works. You can add features later, but you can't easily remove complexity.

The progression:

  • Level 1: Excel/Google Sheets - if it solves the problem, stop here!
  • Level 2: Python script - for automation and data processing
  • Level 3: Streamlit dashboard - when you need visualization
  • Level 4: React app - when you need interactivity
  • Level 5: Full web app with backend - when you need users, auth, databases
Analogy: Don't buy a semi-truck to get groceries. Match the tool to the job.
Start Fresh Chats for New Tasks AI Workflow
AI has a "context window" - it can only remember so much. Long conversations get confused. Start fresh for new topics.

When to start a new chat:

  • Switching to a completely different task
  • The AI seems confused or gives wrong answers
  • You've been chatting for 30+ minutes
  • Starting a new file or feature

When to continue the same chat:

  • Follow-up questions on the same topic
  • Iterating on code the AI just wrote
  • Debugging something the AI created
Analogy: It's like a whiteboard - eventually it gets too cluttered to read. Erase it and start fresh.
Give Context, Get Better Results AI Workflow
The AI doesn't know what you know. Share relevant files, explain your goal, describe your constraints. More context = better code.

Good context to provide:

  • What you're trying to accomplish (the goal)
  • Who will use this (audience/users)
  • What technologies you're already using
  • Constraints (must work offline, must use Python, etc.)
  • Example data or sample files
Analogy: Imagine hiring a contractor without telling them the budget, timeline, or what room you're renovating. The more they know, the better they deliver.
Verify AI Output - Trust but Verify AI Workflow
AI can "hallucinate" - confidently make things up. Always test the code, verify the facts, and don't blindly trust output.

Things to verify:

  • Code: Run it! Does it actually work?
  • Numbers: Do the calculations check out?
  • Links/URLs: Do they actually exist?
  • Facts: Is this information current and accurate?
  • Best practices: Is this how things are actually done?
Analogy: AI is like a very confident intern - helpful and fast, but you still need to review their work.
Keep Files Small and Focused Code Organization
One file should do one thing well. If a file grows past 200-300 lines, consider splitting it. This makes code easier to understand and maintain.

Signs a file is too big:

  • You have to scroll a lot to find things
  • The file does many unrelated things
  • Multiple people need to edit it at once
  • It's hard to describe what the file does in one sentence

How to split:

  • Group related functions into their own file
  • Separate data processing from display logic
  • Put reusable utilities in a shared file
Analogy: A junk drawer is fine until you need to find something. Organized drawers are slower to set up but faster to use.
Use Clear, Descriptive Names Code Organization
Name things for what they do, not what they are. calculate_monthly_revenue() is better than process_data(). Future you will thank present you.

Naming guidelines:

  • Variables: Nouns that describe the data (user_count, monthly_sales)
  • Functions: Verbs that describe the action (get_user(), calculate_total())
  • Files: Describe the contents (data_processing.py, chart_utils.py)
  • Avoid: Single letters (x), abbreviations (proc_dat), generic names (stuff)
Analogy: Imagine labeling every box in your house "stuff". Good luck finding anything!
Comment the Why, Not the What Code Organization
Good code is self-explanatory for what it does. Comments should explain why - the business logic, edge cases, or non-obvious decisions.

Bad comments (explain what):

  • # Add 1 to counter - we can see that!
  • # Loop through items - obvious from the code

Good comments (explain why):

  • # Add 1 because the API uses 1-based indexing
  • # Skip items with no sales - they break the chart
  • # Business rule: discounts only apply to orders over $100
Analogy: A recipe doesn't need to say "put the pan on the stove" - that's obvious. But "use low heat to prevent burning" is useful context.
Test As You Go Testing
Don't write 500 lines then test. Write a little, test it, write more. Finding bugs in 20 lines is easier than finding them in 500.

Simple testing approach:

  • Write one function, run it with sample data
  • Add print statements to see what's happening
  • Test edge cases: empty data, weird values, large numbers
  • For dashboards: click every button, try every filter
Analogy: Taste the soup while cooking, not just at the end. Easier to fix while you're making it.
Keep Sample Data Handy Testing
Always have small test files ready. A 100-row CSV is easier to debug than a 1-million-row dataset. Create sample data that covers edge cases.

Good sample data includes:

  • Normal, expected values
  • Empty or missing values
  • Extreme values (very large, very small, negative)
  • Edge cases specific to your data (e.g., leap years, holidays)
  • At least one example of each "type" in your data
Analogy: Crash test cars don't test with a real family inside. Use small, safe test data first.
Choose the Right Chart Type Data Visualization
The chart type should match your message. Don't use a pie chart for everything - each chart type has a purpose.

Chart selection guide:

  • Bar chart: Comparing categories (sales by region)
  • Line chart: Trends over time (monthly revenue)
  • Pie chart: Parts of a whole (market share) - use sparingly!
  • Scatter plot: Relationship between two variables
  • Table: When exact numbers matter more than patterns
Analogy: You wouldn't use a hammer to screw in a lightbulb. Use the right tool for the job.
Lead with the Insight Data Visualization
Don't make people hunt for the point. Use titles that state the insight, not just describe the data. "Sales Up 23%" beats "Q4 Sales Data".

Tips for clear visualizations:

  • Title states the takeaway, not just the topic
  • Highlight the most important number or trend
  • Remove clutter - if it doesn't help, remove it
  • Use color intentionally (red for bad, green for good)
  • Add context - is 23% growth good or bad for this metric?
Analogy: Newspaper headlines tell you the news upfront. "Fire Destroys Building" not "Building Temperature Data".
